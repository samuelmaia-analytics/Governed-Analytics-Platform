# Metric Catalog

## Objective

This catalog defines the business metrics that should be treated as canonical across Streamlit, Power BI, SQL and future semantic assets.

## Usage rules

- Metrics should be defined once and reused across consumers.
- Streamlit and Power BI should consume the same published semantics whenever possible.
- The internal analytical layer may support engineering and enrichment, but the executive metric contract should come from governed published assets.
- Each metric must have explicit grain, owner and interpretation notes.

## Recommended semantic usage

### Metrics that should remain shared across all executive consumers

- `revenue_gross`
- `orders`
- `customers`
- `avg_ticket`
- `delay_rate`
- `avg_delivery_time_days`
- `avg_review_score`

### Metrics best served from semantic marts

- `active_sellers`
- `seller_delay_rate`
- `seller_avg_delivery_days`
- `retention_m1`
- `retention_m3`

## Interpretation notes

- `revenue_gross` in this project includes item price plus freight. This must stay explicit to avoid comparison drift.
- `avg_ticket` must always be calculated at order level, not item level.
- `delay_rate` should be interpreted over delivered population when comparing logistics performance.
- `avg_review_score` may contain nulls because not every order has a review.
- `active_sellers` is an operational activity metric, not a master-data count.

## Consumption contract

- Streamlit should display and format these metrics, but should progressively reduce metric-definition ownership in app code.
- Power BI should import datasets derived from the same published semantics.
- SQL examples in the repository should reference the same formulas and labels where possible.
